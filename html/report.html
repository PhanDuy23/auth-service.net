Dependency Injection (DI)

DI là kỹ thuật cung cấp các phụ thuộc từ bên ngoài thay vì để đối tượng tự khởi tạo. ASP.NET Core có container DI tích hợp sẵn.

❌ Không dùng DI — Tight Coupling

private EmailService _email = new EmailService();
  • Khó thay thế EmailService
  • Không thể mock khi viết Unit Test
  • UserService phụ thuộc trực tiếp vào class cụ thể

✅ Dùng DI — Loose Coupling

private readonly IEmailService _email; public AuthService(IEmailService email) { _email = email; }
  • Dễ thay bằng MockEmailService khi test
  • Container tự inject — không cần new

Trong project này, tất cả service đều được đăng ký ở ServiceCollectionExtensions.cs:

builder.Services.AddScoped<IAuthService, AuthService>(); builder.Services.AddScoped<IRegistrationService, RegistrationService>(); builder.Services.AddScoped<ITwoFactorService, TwoFactorService>(); builder.Services.AddScoped<IPasswordService, PasswordService>(); builder.Services.AddScoped<IUserService, UserService>(); builder.Services.AddScoped<IAdminService, AdminService>();
AddScoped

Tạo một instance mới cho mỗi HTTP request. Phù hợp với service cần truy cập DB.

AddSingleton

Tạo một instance duy nhất cho toàn bộ ứng dụng. Phù hợp với config, cache.

AddTransient

Tạo instance mới mỗi lần inject. Phù hợp với service nhẹ, không có trạng thái.

Async / Await

Cơ chế bất đồng bộ giúp server không bị block thread khi chờ I/O (database, email, API bên ngoài). Toàn bộ service trong project dùng pattern này.

async

Đánh dấu method có thể chứa thao tác bất đồng bộ. Method luôn trả về Task hoặc Task<T>.

await

Tạm dừng execution và nhường thread. Khi tác vụ hoàn tất, method tiếp tục từ điểm đó.

Ví dụ thực tế trong AuthService.cs:

public async Task<...> LoginAsync(LoginDto dto) { // Không block thread khi query DB var user = await _userManager.FindByEmailAsync(dto.Email); // Không block thread khi check password (hashing) var valid = await _userManager.CheckPasswordAsync(user, dto.Password); // Không block thread khi ghi DB await _userManager.UpdateAsync(user); }

Role: Customer | Employee | Admin Email Confirmation Rollback nếu gửi mail thất bại

Kiểm tra role hợp lệ (Customer / Employee / Admin). Kiểm tra email chưa tồn tại trong DB.

Tạo ApplicationUser

Gán Role

Xác nhận email

Đăng nhập (Login)

Endpoint: POST /api/auth/login — Xử lý bởi AuthService.LoginAsync

1
Tìm user theo email
Nếu không tìm thấy hoặc IsActive = false → trả 401 "Email hoặc mật khẩu không đúng" (không tiết lộ tài khoản có tồn tại).
2
Kiểm tra Lockout
Nếu đang bị khóa → trả 423 kèm số phút còn lại. Identity tự động tăng AccessFailedCount mỗi lần sai mật khẩu.
3
Kiểm tra mật khẩu
CheckPasswordAsync — hash và so sánh. Sai → AccessFailedAsync. Nếu vượt ngưỡng → auto lockout.
4
Kiểm tra email đã xác nhận
Nếu chưa confirm email → trả 400. Reset AccessFailedCount sau khi mật khẩu đúng.
5a
Có bật 2FA → Pending Token
Tạo pendingToken, set vào HttpOnly cookie. Response trả requiresTwoFactor: true. Client phải gọi tiếp POST /api/auth/2fa/verify.
5b
Không có 2FA → Cấp token
Sinh Access Token (JWT, 30 phút) và Refresh Token (24 giờ). Refresh token lưu cookie HttpOnly; Secure; SameSite=Strict.
Access Token (JWT)

Chứa userId, email, roles. Hạn: 30 phút. Gửi kèm mỗi request qua Authorization: Bearer ...

Refresh Token

Chuỗi ngẫu nhiên, hash SHA-256 lưu DB. Hạn: 24 giờ. Lưu trong HttpOnly cookie — JavaScript không đọc được.

Remember Me

Nếu rememberMe: true → cookie có Expires cụ thể. Nếu false → session cookie, mất khi đóng browser.

07

Two-Factor Authentication (2FA)

Dùng chuẩn TOTP (RFC 6238) — tương thích Google Authenticator, Microsoft Authenticator. Secret key 160-bit, mã OTP 6 chữ số, chu kỳ 30 giây.

🔧 Bật 2FA

1
GET /api/auth/2fa/setup
Tạo secret key tạm (Base32) + QR URI. Chưa lưu DB.
2
Quét QR vào Authenticator app
App sinh mã OTP 6 chữ số mỗi 30 giây.
3
POST /api/auth/2fa/enable
Gửi secretKey + code. Server xác minh OTP → lưu secret vào DB, bật TwoFactorEnabled = true.
4
Nhận 8 Recovery Codes
Chỉ hiển thị một lần duy nhất. Dùng khi mất thiết bị xác thực.

🔐 Đăng nhập với 2FA bật

1
POST /api/auth/login
Mật khẩu đúng → server set Pending Token vào HttpOnly cookie, trả requiresTwoFactor: true.
2
POST /api/auth/2fa/verify
Gửi mã OTP 6 chữ số. Server đọc pending token từ cookie, verify TOTP (window ±1 step).
✓
Nhận Access Token + Refresh Token
Pending token bị xóa sau khi dùng. Refresh token cookie được set.
R
Dùng Recovery Code
POST /api/auth/2fa/verify-recovery. Sau khi dùng → 2FA tự động tắt, cần cấu hình lại.

⚠️ Recovery Code — hành vi đặc biệt

Sau khi dùng recovery code thành công: TwoFactorEnabled = false, secret key bị xóa, tất cả recovery codes còn lại bị revoke. Response có cờ requiresTwoFactorReconfiguration: true để frontend nhắc người dùng bật lại 2FA.

08

Passkey — Xác thực thế hệ mới

🔑
WebAuthn / FIDO2 W3C Standard

Không mật khẩu. Không OTP. Chỉ cần vân tay hoặc Face ID.

Passkey là phương thức xác thực dựa trên mật mã khóa công khai (asymmetric cryptography). Thay vì gõ mật khẩu, thiết bị của người dùng tạo một cặp khóa — private key lưu an toàn trên thiết bị, public key gửi lên server. Khi đăng nhập, thiết bị ký một challenge bằng private key; server xác minh bằng public key. Private key không bao giờ rời khỏi thiết bị.

❌ Password
  • Dễ bị đánh cắp qua phishing
  • Dễ bị brute-force
  • Người dùng hay đặt yếu hoặc tái dùng
  • Server phải lưu hash — rủi ro nếu DB bị lộ
⚠️ Password + 2FA (TOTP)
  • Tốt hơn, nhưng vẫn có mật khẩu
  • OTP có thể bị đánh cắp realtime qua AiTM attack
  • Người dùng cần cài thêm app
  • Project hiện tại đang ở mức này
✅ Passkey
  • Phishing-resistant — không có gì để đánh cắp
  • Không cần nhớ mật khẩu
  • Xác thực bằng biometrics / PIN thiết bị
  • Tích hợp sẵn trên iOS, Android, Windows Hello

📋 Luồng đăng ký Passkey

1
Server tạo Registration Challenge
Trả về challenge (random bytes), rpId (domain), userId. Challenge có thời hạn ngắn, dùng một lần.
2
Browser gọi navigator.credentials.create()
OS hiện dialog xác nhận bằng vân tay / Face ID / PIN. Thiết bị tạo cặp khóa public/private.
3
Gửi public key + attestation lên server
Server xác minh attestation, lưu credentialId và publicKey vào DB. Private key không rời thiết bị.
✓
Passkey đã đăng ký thành công
Lần sau đăng nhập không cần mật khẩu.

🔐 Luồng đăng nhập bằng Passkey

1
Server tạo Authentication Challenge
Trả về challenge mới và danh sách allowedCredentials của user.
2
Browser gọi navigator.credentials.get()
OS xác thực người dùng qua biometrics. Thiết bị ký challenge bằng private key.
3
Gửi signature lên server
Server tra credentialId, lấy public key từ DB, xác minh chữ ký. Đúng → đăng nhập thành công.
✓
Cấp Access Token + Refresh Token
Tương tự flow login thông thường — cùng cơ chế JWT/cookie.

🛡️ Tại sao Passkey chống phishing hoàn toàn?

Khi tạo passkey, rpId (domain của website) được gắn vào credential. Khi đăng nhập, thiết bị tự kiểm tra domain hiện tại có khớp với rpId không.

Nếu user bị lừa vào fake-bank.com thay vì bank.com — thiết bị từ chối ký vì domain không khớp. Không có credential nào được gửi đi.

So sánh với TOTP

TOTP (Google Authenticator) vẫn có thể bị đánh cắp realtime — attacker dựng trang giả, lấy OTP ngay khi user nhập và dùng lại trong vài giây. Passkey không có thông tin nào để đánh cắp.

Triển khai trong ASP.NET Core
  • Thư viện phổ biến: Fido2NetLib
  • Lưu credentialId, publicKey, signCount vào DB
  • 2 endpoint: POST /passkey/register và POST /passkey/login
  • Có thể kết hợp với Identity: một user có nhiều passkeys
Hỗ trợ trình duyệt & nền tảng
Chrome 108+ Safari 16+ Firefox 122+ iOS 16+ Android 9+ Windows Hello iCloud Keychain Google Password Manager

Passkey được đồng bộ qua cloud (iCloud / Google) — mất điện thoại không mất passkey.

Token & Cookie

?
Câu hỏi

Tại sao chọn JWT thay vì Session truyền thống?

🗄️ Session truyền thống

  • Server phải lưu session trong memory / Redis cho mỗi user
  • Không scale ngang dễ — request phải đến đúng server giữ session
  • Phù hợp monolith, khó dùng cho microservices hay mobile API
  • Cần sticky session hoặc session store dùng chung (Redis)

🎫 JWT (Stateless)

  • Server không lưu gì — mọi thông tin trong token, verify bằng chữ ký
  • Scale ngang tự do — bất kỳ instance nào cũng xác minh được
  • Phù hợp REST API, microservices, mobile, SPA
  • Payload chứa sẵn claims — không cần query DB mỗi request

Đánh đổi: JWT không thể thu hồi ngay lập tức trước khi hết hạn (vì stateless). Project giải quyết bằng cách giữ Access Token ngắn hạn (30 phút) và dùng Refresh Token lưu DB để có thể revoke khi cần.

?
Câu hỏi

Nguyên tắc set thời hạn cho mỗi loại token?

Thời hạn token là cân bằng giữa bảo mật (token ngắn → ít rủi ro nếu bị lộ) và UX (token dài → ít phải login lại). Không có con số tuyệt đối — tuỳ ngữ cảnh ứng dụng.

Access Token — 30 phút
  • Stateless → không thu hồi được
  • Nếu bị đánh cắp, thiệt hại giới hạn trong 30 phút
  • Ngắn hơn (5–15 phút) nếu ứng dụng tài chính, y tế
  • Dài hơn (1–2 giờ) nếu internal tool ít nhạy cảm
Refresh Token — 24 giờ
  • Lưu DB → có thể revoke ngay
  • Luân chuyển mỗi lần dùng (Refresh Token Rotation)
  • 7–30 ngày nếu có "Remember Me"
  • Lưu HttpOnly cookie — JS không đọc được
Pending / Special Tokens
  • 2FA Pending: 5–10 phút — chỉ dùng 1 lần
  • Email confirm: 24 giờ — user có thời gian mở mail
  • Reset password: 15–60 phút — nhạy cảm, nên ngắn
  • OAuth state: vài phút — chỉ dùng 1 lần chống CSRF

Quy tắc thực tế:

Token càng khó revoke → càng nên ngắn. Access Token stateless không revoke được → 30 phút. Refresh Token có DB → có thể dài hơn nhưng phải rotation.

Token dùng 1 lần → hết hạn ngay sau khi dùng. Pending 2FA, reset password không bao giờ được dùng lại dù chưa hết hạn thời gian.

Loại Nội dung Thời hạn Lưu trữ Mục đích
Access Token (JWT) userId, email, roles, exp 30 phút In-memory JS (không lưu localStorage) Xác thực mỗi API request
Refresh Token Chuỗi ngẫu nhiên, hash SHA-256 lưu DB 24 giờ HttpOnly Cookie Cấp lại Access Token khi hết hạn
Pending Token Chuỗi ngẫu nhiên, hash lưu DB 5 phút HttpOnly Cookie Xác thực bước 2 của 2FA login
Email Confirm Token Token do ASP.NET Identity tạo 1 ngày (mặc định) URL link trong email Xác nhận email khi đăng ký
Password Reset Token Token do ASP.NET Identity tạo 1 giờ URL link trong email Reset mật khẩu qua email

Luồng Refresh Token

Khi Access Token hết hạn, client gọi POST /api/auth/refresh-access-token — server đọc refresh token từ HttpOnly cookie, tìm trong DB, kiểm tra còn hiệu lực, cấp Access Token mới.

// CookieService đọc từ HttpOnly cookie — client không cần gửi gì var rawToken = _cookieService.GetRefreshToken(); // Hash và tìm trong DB, kiểm tra IsActive và User.IsActive var tokenHash = TokenService.HashToken(rawToken); var stored = await _db.RefreshTokens .Include(t => t.User) .FirstOrDefaultAsync(t => t.TokenHash == tokenHash);

External Login (GitHub / Google)

GitHub OAuth 2.0

1
Frontend redirect đến GitHub
Tạo state ngẫu nhiên (chống CSRF), lưu sessionStorage. Redirect đến github.com/login/oauth/authorize với scope: user:email.
2
GitHub callback với code
GitHub redirect về github-callback.html kèm code và state. Frontend verify state rồi POST code lên server.
3
POST /api/auth/github/callback
Server đổi code → GitHub access token → lấy profile. Nếu email ẩn → gọi thêm /user/emails để lấy primary verified email.
4
Tìm / tạo / link user
Tìm theo AspNetUserLogins (GitHub ID) → tìm theo email → tạo mới. Gán role User cho tài khoản mới. EmailConfirmed = true vì GitHub đã xác thực.
5
Cấp JWT + Refresh Token
Tương tự login thường. Access token trả về response, refresh token set vào HttpOnly cookie.

Google OAuth 2.0 + OpenID Connect

1
Redirect đến /api/auth/google/login
Server tạo challenge redirect. ASP.NET Core xử lý OAuth flow nội bộ.
2
Google callback về server
Server nhận code, đổi lấy id_token + access token, lấy profile từ OpenID Connect.
3
Tìm / tạo / link user
Tương tự GitHub. Email đã verified từ Google. Link vào AspNetUserLogins.
4
Set oauth_access_token cookie
Server set JWT vào cookie tạm, redirect về login.html. Frontend đọc cookie, xóa ngay, lưu token vào biến JS.

Authorization — Phân quyền

Project dùng kết hợp Role-based, Claims-based, Policy-based và Resource-based authorization.

Loại Dựa trên Ví dụ trong project Endpoint
Role-based Role của user [Authorize(Roles = "Admin")] GET /api/admin/users
Claims-based Claim permission trong JWT p.RequireClaim("permission", Permissions.UsersRead) GET /api/users
Policy-based Claims-based Requirement được đặt tên [Authorize(Policy = Policy.GetUser)] GET /api/users
Resource-based User ID so với resource SameUserRequirement — chỉ sửa profile của chính mình PUT /api/users/{id}/profile
users.read

Gán cho Employee và Admin. Cho phép đọc danh sách user qua GET /api/users.

users.delete

Chỉ gán cho Admin. Cho phép xóa (soft delete) user qua DELETE /api/admin/users/{id}.

profile.edit

Dùng trong Resource-based policy SameUserRequirement, không kiểm tra claim mà so sánh ID.

Đăng ký policy từ claim trong AuthorizationExtensions.cs

AddAuthorizationBuilder() .AddPolicy(Permissions.UsersRead, p => p.RequireClaim("permission", Permissions.UsersRead)) .AddPolicy(Permissions.UsersDelete, p => p.RequireClaim("permission", Permissions.UsersDelete)) .AddPolicy(Permissions.ProfileEdit, p => p.AddRequirements(new SameUserRequirement()));
Policy

Policy-based & Resource-based

Policy: UsersRead

// Controller dùng attribute [Authorize(Policy = Permissions.UsersRead)] [HttpGet("users")] public async Task<IActionResult> GetUsers() { }

JWT phải chứa claim permission = "users.read" — chỉ Employee và Admin có.

Resource-based: SameUserRequirement

// Gọi thủ công trong controller var result = await _authorizationService .AuthorizeAsync( User, id, Permissions.ProfileEdit); if (!result.Succeeded) return StatusCode(403, ...);

SameUserHandler so sánh sub trong JWT với id trên URL — chỉ chính chủ mới được sửa.

Admin endpoints — chỉ dành cho role Admin

GET /api/admin/users GET /api/admin/users/{id} GET /api/admin/employees PATCH /api/admin/users/{id}/lock PATCH /api/admin/users/{id}/unlock
?
Câu hỏi quan trọng

Khi nào nên lấy claims từ JWT? Khi nào nên lấy từ database?

✅ Lấy từ JWT khi…

  • Dữ liệu ít thay đổi — role, permission, userId, email
  • Cần hiệu năng cao — không mất 1 round-trip vào DB cho mỗi request
  • Hệ thống stateless — microservice, API gateway cần tự xác minh token
  • Thông tin chỉ cần đúng tại thời điểm đăng nhập — không cần realtime

Trong project này:

sub, email, role, permission, fullName — đọc thẳng từ User.Claims trong controller, không query DB.

⚠️ Lấy từ database khi…

  • Cần dữ liệu realtime — token chưa hết hạn nhưng user đã bị khóa (IsActive = false)
  • Quyền vừa bị thu hồi — đổi role, xóa permission sau khi cấp token
  • Thao tác nhạy cảm — đổi mật khẩu, xóa tài khoản, thanh toán
  • Cần dữ liệu đầy đủ không có trong token — avatar, dateOfBirth, department

Trong project này:

GET /api/me lấy userId từ JWT rồi gọi _userManager.FindByIdAsync() để lấy profile mới nhất từ DB, không tin hoàn toàn vào fullName trong token.

Nguyên tắc chung: JWT phù hợp cho authorization (ai được làm gì). Database cần thiết cho state thực tế của user (họ còn active không, dữ liệu mới nhất là gì). Token ngắn hạn (30 phút) giúp thu hẹp khoảng lệch giữa hai nguồn.
